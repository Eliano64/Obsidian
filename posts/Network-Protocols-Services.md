# 实体和对等实体

实体是任何可以发送或接收信息的硬件或软件进程。

例子：

- 网卡可以发送和接收信息，因此可以看作硬件实体。
- 浏览器进程、Web 服务器进程可以发送和接收应用报文，因此可以看作软件实体。

对等实体是通信双方相同层次中的实体。例如，发送主机应用层中的浏览器进程和接收主机应用层中的 Web 服务器进程互为对等实体；两端数据链路层中的网卡相关功能也可以互为对等实体。

# 协议

协议是控制**两个对等实体**进行**逻辑通信**的规则集合。

两个实体之间并不是有物理链接。对等层看起来像在直接通信，实际数据仍会沿发送方协议栈向下、经过网络传输、再沿接收方协议栈向上。

协议三要素：**语法、语义、同步**

| 要素       | 含义                        | 例子                        |
| -------- | ------------------------- | ------------------------- |
| 语法       | 数据与控制信息的结构或格式             | 首部字段如何排列、字段占多少位           |
| 语义$^{1}$ | 需要发出何种控制信息，完成何种动作以及做出何种响应 | 收到 HTTP GET 后应返回请求资源或错误状态 |
| 同步       | 事件实现顺序的详细说明               | 先发送请求，再接收响应；连接建立后再传输数据    |

$^{1}$在协议中，一个控制信息的含义往往就是通过它触发的行为来定义的。所以语义也可以理解为某协议字段的**含义及处理规则**。

> [!note] 协议看横向
> 协议描述的是同一层、不同系统中的对等实体如何**逻辑通信**。它解决的是横向规则问题。

# 服务和接口

服务是下层通过层间接口向上层提供的功能。

接口是同一系统内相邻两层之间交互的位置。上层通过[[#服务访问点和服务原语|服务访问点 SAP]] 使用下层服务。

![900](../assets/protocol-service-interface-relation.svg)

第 $n$ 层通过第 $n-1$ 层提供的服务完成自己的通信功能，同时又向第 $n+1$ 层提供服务。也就是说，服务关系是逐层向上的：低层提供能力，高层使用能力。

这三个概念的方向不同：

| 概念  | 方向          | 关键问题           |
| --- | ----------- | -------------- |
| 协议  | 横向，同层对等实体之间 | 对等实体按什么规则通信    |
| 服务  | 纵向，下层提供给上层  | 下层能为上层做什么      |
| 接口  | 本机内部，相邻层之间  | 上层在哪里、如何访问下层服务 |

# 服务访问点和服务原语

服务访问点 SAP 是上层访问下层服务的位置。可以把它理解为层间服务的入口。

服务原语是描述层间服务请求和响应的抽象操作。它强调的是同一系统内上下层如何交互，而不是两个主机之间的物理传输细节。



一个 SAP 可以使用多种服务原语，同一种原语也可以出现在不同 SAP 上。原语的参数通常还要指出本地或远端 SAP，从而确定服务关系的两个端点。

# PDU、SDU、IDU 和 ICI

对等层之间传送的数据单元称为**协议数据单元** PDU。

上层交给下层、由下层服务负责传送的有效数据称为**服务数据单元** SDU。SDU 是下层要处理的载荷，不包含下层自己添加的协议控制信息。

同一系统内，相邻层实体通过接口实际传递的整体称为**接口数据单元** IDU。IDU 除了包含 SDU，还可以携带只供层间接口使用的**接口控制信息** ICI：

$$
\text{IDU}=\text{ICI}+\text{SDU}
$$

| 数据单元 | 所在关系          | 作用                            |
| ---- | ------------- | ----------------------------- |
| PDU  | 同层对等实体之间的逻辑通信 | 本层协议处理和传送的数据单元                |
| SDU  | 上层交给下层的有效数据   | 作为下层封装时的载荷                    |
| IDU  | 本机相邻层接口之间     | 接口实际传递的整体                     |
| ICI  | 本机相邻层接口之间     | 告诉下层如何处理 SDU，不随 PDU 发送给远端对等实体 |
|      |               |                               |

![760](../assets/pdu-sdu-relation.svg)

在封装过程中，上层交给本层的数据就是本层的 SDU。本层添加首部或尾部后，形成用于本层对等实体逻辑通信的 PDU。 ^23b2dc

若用 $n$ 表示当前层，则封装关系为：

$$
n\text{-PDU}=n\text{-PCI}+n\text{-SDU}
$$

其中 PCI 是本层协议控制信息，例如首部或尾部。相邻层的数据内容通常满足：

$$
(n+1)\text{-PDU}=n\text{-SDU}
$$

这个等式描述的是**数据内容如何成为下一层的载荷**，**但接口上不只是传递 SDU**。上层实体通过 SAP 调用下层服务时，**接口上传递的是 IDU**；低层从 IDU 中取得 SDU，并按 ICI 的要求处理它。

例如：

- HTTP 请求交给 TCP 后，对 TCP 来说是 SDU。
- TCP 加上 TCP 首部后，形成 TCP 报文段，这是运输层 PDU。
- TCP 报文段交给 IP 后，又成为网络层的 SDU。
- IP 加上 IP 首部后，形成 IP 数据报，这是网络层 PDU。

**多个 SDU 可以合成为一个 PDU，一个 SDU 也可以拆分为多个 PDU**。具体如何处理取决于该层协议设计。

> [!warning] 区分接口传递单位和数据内容对应关系
> 当相邻高层实体把一个数据单元通过接口交给低层实体时，接口实际传递的是 **IDU**；低层把其中需要传送的有效数据视为自己的 **SDU**。
> 
> 但“高层 PDU 到下一层后成为了低层 SDU”的说法也是正确的。两种说法并不矛盾，它们分别描述接口形式和数据内容。

# 封装、转发与解封

一次网页访问中，数据会沿发送主机协议栈向下封装，经过路由器转发，再沿接收主机协议栈向上解封。

[html-card height=760](../assets/network-encapsulation-decapsulation-slides.html)

发送主机从上到下封装：

1. 应用层构造 HTTP 请求报文。
2. 运输层添加 TCP 首部，形成 TCP 报文段。
3. 网络层添加 IP 首部，形成 IP 数据报。
4. 数据链路层添加帧首部和帧尾，形成帧。

最后物理层把帧看作比特流，并转换成信号发送。**物理层不参与数据封装**。

接收主机从下到上解封：

1. 物理层把信号恢复为比特流。
2. 数据链路层识别帧，去掉帧首部和帧尾，取出 IP 数据报。
3. 网络层去掉 IP 首部，取出运输层数据。
4. 运输层去掉 TCP 首部，取出应用报文。
5. 应用层把 HTTP 报文交给对应应用进程处理。

> [!important] 数据封装的核心
> 每一层把上层交来的数据整体当作自己的数据部分，再加上本层控制信息。接收端按相反顺序逐层去掉控制信息。即[[#^23b2dc|上文所示]]
