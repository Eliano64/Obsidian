# 中断响应与处理过程

一次中断处理可以分成两个阶段：

- **硬件阶段**：CPU 完成中断响应，也常称为**中断隐指令**完成的工作。它会暂停现行程序，保存返回所需的断点信息，并把控制权转入操作系统内核中的处理程序入口。
- **软件阶段**：操作系统内核或设备驱动执行中断服务程序。

[html-card height=760](../assets/exception-interrupt-handling-slides.html)

中断处理的主线：

1. 外部中断源提出中断请求。
2. CPU 在当前指令执行结束后检查中断请求。
3. 若满足响应条件，CPU 进行中断判优，选出要响应的中断源。
4. CPU 执行中断隐指令：关中断、保存断点、切换到内核态，并引出内核中的中断服务程序。
5. 内核态中断服务程序保存现场。
6. 内核态中断服务程序完成具体服务。
7. 中断服务程序恢复现场。
8. 执行中断返回指令，回到原程序断点继续执行。

> [!note] 什么时候重新开中断？
> 单重中断：中断返回指令会恢复中断前的 CPU 状态（包括中断允许位），因此开中断发生在步骤 8；处理期间全程关中断。
> 多重中断：在步骤 5–7 之间由软件显式开中断，使更高优先级的中断源可以打断当前服务程序。详见 [[#单重中断与多重中断]]。

> [!important] 中断隐指令不是指令
> 中断隐指令不是指令系统里显式写出来的机器指令，而是由**硬件**自动完成的一组操作：关中断、保存断点、切换到内核态、引出中断服务程序。
> 后续的处理程序在操作系统内核中执行。

# 中断隐指令

中断隐指令的主要任务：

| 任务       | 作用                                    |
| -------- | ------------------------------------- |
| 关中断      | 防止保存断点、引出处理程序期间又被新的中断打断               |
| 保存断点     | 保存原程序的 PC，使中断处理完成后能返回原程序              |
| 转入内核态    | 使 CPU 从普通程序的执行状态转入内核态，执行内核代码、访问内核数据结构 |
| 引出中断服务程序 | 根据中断类型号或中断向量找到内核中的服务程序入口，并送入 PC       |
|          |                                       |


> [!important] 保存断点和保护现场不是同一件事
> | 概念 | 保存内容 | 主要由谁完成 |
> | --- | --- | --- |
> | 保存断点 | PC，即原程序下一步应从哪里继续 | 中断隐指令(硬件) |
> | 保护现场 | 通用寄存器、状态字、可能被服务程序破坏的运行环境 | 中断服务程序(软件) |

# 中断向量表

中断发生后，CPU 必须立即转入对应的内核入口（即**中断向量**），没有时间临时搜索驱动程序。

解决办法是提前建立一张按编号索引的**中断向量表**：启动阶段先把表准备好，设备初始化时建立向量号与驱动的关系，中断发生时 CPU 只负责按号查表。

| 对象            | 含义                   |
| ------------- | -------------------- |
| 中断向量号或类型号 $i$ | 标识异常或中断类型，也是表索引      |
| 向量表项地址        | 第 $i$ 个表项在内存中的位置     |
| 向量表项          | 保存处理程序入口信息及必要控制属性的记录 |
| 中断向量          | 表项给出的处理程序入口地址        |
| 中断服务程序        | 入口最终转入的内核代码          |

## 启动阶段

| 参与者 | 负责的内容 |
| --- | --- |
| 处理器体系结构 | 规定向量号范围、表项格式、查表方式和表基址寄存器 |
| 固件 / BIOS | 在操作系统尚未运行时提供启动期临时表和基础处理入口 |
| 操作系统内核 | 建立最终向量表，填写异常入口、默认入口和中断公共入口 |
| 中断控制器与驱动 | 建立设备 IRQ、向量号和具体驱动处理函数之间的运行时关系 |


机器刚上电时，操作系统还没有运行，但启动过程仍可能需要异常处理和基本 I/O。固件会先准备一套早期中断入口。

内核接管机器后，不会把固件提供的临时表直接当作最终运行环境，而是建立与自身内核代码、保护机制和驱动框架配套的表。顺序是：

1. 暂时屏蔽或严格控制外部中断，只保留启动所需的最小执行环境；
2. 准备异常入口、中断公共入口和默认处理程序；
3. 分配向量表，并预先创建体系结构规定数量的表项；
4. 把处理程序入口信息和控制属性填入相应表项，未使用的表项通常指向默认处理程序或保持为不可用状态；
5. 通过专用指令把表的基址和界限装入 CPU 的向量表寄存器。

此时 CPU 已经知道最终向量表位于哪里，但设备中断还不应贸然开放。内核最终表建立前仍可能出现处理器异常，因此启动代码必须先安装最小可用的早期异常入口；外部设备中断则通常保持屏蔽，等中断控制器和驱动准备完成后再开放。

## 运行准备

最终向量表建立后，还需要把“某个设备发出请求”连接到“某个驱动函数”。以设备驱动初始化为例，过程通常是：

1. 驱动向内核申请中断资源并登记处理函数；
2. 内核为设备选择或分配向量号；
3. 内核在自己的第二级分派表中记录向量号与驱动处理函数的关系；
4. 内核配置 PIC、APIC 等中断控制器，把设备 IRQ 映射到该向量号；
5. 所有入口和映射就绪后，内核解除该中断的屏蔽。

硬件向量表的槽位通常在启动时已经按体系结构规定的数量建立。表项可能直接指向某个处理程序，也可能统一指向一组汇编入口，再由内核根据向量号进行第二次分派。共享中断还可能对应一条驱动处理链。

因此，注册新的中断类型或设备处理程序时，系统主要是在**分配向量号、更新表项或登记第二级分派关系**，并不是到这时才临时创建一个向量表槽位。

## 中断到达

运行时的基本过程是：

1. 处理器内部异常逻辑确定异常向量号，或者中断控制器向 CPU 提供外部中断向量号；
2. CPU 从专用寄存器取得向量表基址；
3. CPU 用向量号定位相应表项，读取处理程序入口和体系结构要求的控制信息；
4. CPU 按规定保存必要现场并转入内核入口；
5. 内核公共入口进一步分派到异常处理程序或设备驱动的中断处理程序；
6. 处理完成后恢复现场，返回被中断程序。

因此，运行时的数据流是：

$$
\text{向量号}\rightarrow\text{第 }i\text{ 个向量表项}
\rightarrow\text{处理程序入口信息}\rightarrow\text{中断服务程序}
$$

# I/O 中断处理程序

I/O 中断服务程序会直接和 I/O 控制器打交道。一次典型 I/O 完成中断的处理过程：

1. 从控制器读出设备状态。
2. 判断 I/O 是否正常完成。
3. 若异常，按错误原因处理，例如重试、报告错误、唤醒等待进程并返回错误。
4. 若正常，读取数据寄存器或确认 DMA/通道已经完成传输。
5. 更新内核中的 I/O 请求状态和缓冲区状态。
6. 唤醒等待该 I/O 的进程，或把后续工作交给设备驱动、设备独立性软件继续处理。

> [!note]
> 中断处理程序位于操作系统内核中，在 I/O 软件层次里靠近硬件。它负责处理中断应答和硬件状态；更通用的 I/O 调度、设备分配、缓冲区管理通常在设备独立性软件层完成。

# 单重中断与多重中断

![1000](../assets/single-multiple-interrupt-flow.svg)

**单重中断**是在执行中断服务程序时不响应新的中断请求。它流程简单，但高优先级事件可能必须等待当前服务程序结束。

**多重中断**允许在中断服务程序中重新开中断，使更高优先级的中断源可以打断当前中断服务程序。多重中断需要处理中断优先级、屏蔽字和现场嵌套保存。

多重中断成立的关键条件：

- 中断服务程序中提前设置开中断。
- 高优先级中断源有权打断低优先级中断源。
- 使用屏蔽字控制哪些中断源可以继续申请。
- 每次嵌套都要保存对应现场和屏蔽字，返回时逐层恢复。

## 中断屏蔽字

可屏蔽中断会受中断允许标志和屏蔽字控制。以常见约定为例：

- `IF=1`：允许响应可屏蔽中断。
- `IF=0`：禁止响应可屏蔽中断。
- 屏蔽字中 `1` 表示屏蔽某个中断源，`0` 表示该中断源可以正常申请。

不可屏蔽中断不受 `IF` 控制，通常用于电源故障等紧急事件。

> [!warning] 屏蔽字的常见判断
> 屏蔽字中 `1` 越多，通常表示屏蔽范围越大。每个中断源对应的屏蔽字至少要能屏蔽自身，否则可能在处理中被同源中断反复打断。
