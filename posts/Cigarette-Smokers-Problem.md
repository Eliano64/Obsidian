# 问题模型

吸烟者问题有一个供应者进程和三个吸烟者进程。卷烟需要三种材料：烟草、纸、胶水。

- 吸烟者 1 自带烟草，需要纸和胶水。
- 吸烟者 2 自带纸，需要烟草和胶水。
- 吸烟者 3 自带胶水，需要烟草和纸。
- 供应者每次把两种材料组合放到桌上。
- 拥有第三种材料的吸烟者取走桌上的组合，卷烟并抽完。
- 吸烟者抽完后通知供应者，供应者再放下一组材料。

![900](../assets/cigarette-smokers-model.svg)

这个问题可以看成特殊的生产者-消费者问题：供应者是单生产者，三个吸烟者是三类消费者，桌子是容量为 1 的缓冲区。不同点在于，供应者生产的不是同一种产品，而是三种不同材料组合。

# 同步关系

桌上可能出现三种组合：

| 桌上组合 | 谁能取走 | 对应信号量 |
| --- | --- | --- |
| 纸 + 胶水 | 自带烟草的吸烟者 1 | `offer1` |
| 烟草 + 胶水 | 自带纸的吸烟者 2 | `offer2` |
| 烟草 + 纸 | 自带胶水的吸烟者 3 | `offer3` |

还需要一个 `finish` 信号量，表示某个吸烟者已经取走材料并抽完，供应者可以继续放下一组材料。

| 信号量 | 初值 | 含义 |
| --- | --- | --- |
| `offer1` | `0` | 桌上有组合 1：纸 + 胶水 |
| `offer2` | `0` | 桌上有组合 2：烟草 + 胶水 |
| `offer3` | `0` | 桌上有组合 3：烟草 + 纸 |
| `finish` | `0` | 吸烟者完成一次吸烟 |

这里的同步方向是“前 V 后 P”：

- 供应者先放组合，再 `V(offerX)` 通知对应吸烟者。
- 对应吸烟者执行 `P(offerX)`，拿到材料后卷烟、吸烟。
- 吸烟者完成后 `V(finish)`。
- 供应者执行 `P(finish)`，等完成信号后再进入下一轮。

# 伪代码

```text
semaphore offer1 = 0;    // 桌上有纸 + 胶水
semaphore offer2 = 0;    // 桌上有烟草 + 胶水
semaphore offer3 = 0;    // 桌上有烟草 + 纸
semaphore finish = 0;    // 吸烟者完成一次吸烟

provider 进程：
    while (1) {
	    i <- produce; // 提供三种组合之一
        if i == 0 then
            把纸和胶水放到桌上;
            V(offer1);
        else if i == 1 then
            把烟草和胶水放到桌上;
            V(offer2);
        else
            把烟草和纸放到桌上;
            V(offer3);
        P(finish);
    }

smoker1 进程：
    while (1) {
        P(offer1);
        从桌上取走纸和胶水;
        卷烟并抽掉;
        V(finish);
    }

smoker2 进程：
    while (1) {
        P(offer2);
        从桌上取走烟草和胶水;
        卷烟并抽掉;
        V(finish);
    }

smoker3 进程：
    while (1) {
        P(offer3);
        从桌上取走烟草和纸;
        卷烟并抽掉;
        V(finish);
    }
```

## 为什么不需要单独的 mutex

桌子可以看成容量为 1 的缓冲区。供应者每次放入一个组合后，会立刻执行 `P(finish)` 等待吸烟者完成；在此之前不会继续放第二个组合。

对吸烟者来说，同一时刻 `offer1`、`offer2`、`offer3` 至多只有一个能通过 `P` 操作。只有对应组合的吸烟者能取走材料，其他吸烟者仍阻塞。

因此，桌面访问已经被三个 `offer` 和一个 `finish` 的同步关系串行化。这个特定模型下不需要额外设置 `mutex`。

# 变式：缓冲区容量为 N

下面把桌面推广为能够存放 $N$ 组材料组合的缓冲区。这里假设：**一组两种材料占用一个槽位，消费者可以从缓冲区中取走与自己匹配的任意一组组合，缓冲区不要求严格按 FIFO 顺序取出。**

此时供应者不必等待每一位吸烟者抽完，可以连续生产，直到 $N$ 个槽位全部占满。供应者和三个吸烟者可能同时访问缓冲区，因此容量为 1 时隐含的串行关系已经消失。**因此，与容量为1的情况不同，必须用 `empty` 管理容量，并用 `mutex` 保护缓冲区结构。**

![1000](../assets/cigarette-smokers-buffer-n.svg)

> [!warning] 若缓冲区要求严格 FIFO
> 本节方案把缓冲区视为可按类型取出的集合。如果缓冲区必须严格先进先出，即使 `offer2 > 0`，吸烟者 2 也不能绕过位于队首的其他组合。此时应由统一的分派者检查并取出队首，再通知对应吸烟者；或者接受队首组合造成的队头阻塞。不能只凭三个类型计数信号量任意删除队列中间的元素。

## 信号量

| 信号量 | 初值 | 作用 |
| --- | ---: | --- |
| `empty` | $N$ | 记录缓冲区中的空槽位数 |
| `mutex` | `1` | 互斥保护缓冲区及其内部计数 |
| `offer1` | `0` | 缓冲区中纸 + 胶水组合的数量 |
| `offer2` | `0` | 缓冲区中烟草 + 胶水组合的数量 |
| `offer3` | `0` | 缓冲区中烟草 + 纸组合的数量 |

三个 `offer` 在这里都是**计数信号量**，数值可以大于 1。标准的缓冲区容量模型不再需要 `finish`：供应者关心的是有没有空槽位，而不是某位吸烟者是否已经抽完。

## 容量为 N 的伪代码

```text
semaphore empty = N;
semaphore mutex = 1;
semaphore offer1 = 0, offer2 = 0, offer3 = 0;

provider 进程：
    while (1) {
        k <- produce();

        P(empty);              // 先等待空槽，不能拿着 mutex 等待
        P(mutex);
        向缓冲区放入一组类型 k 的材料;
        V(mutex);
        V(offer[k]);           // 放入完成后再公布这种组合
    }

smoker[k] 进程：
    while (1) {
        P(offer[k]);           // 先确认存在自己需要的组合
        P(mutex);
        从缓冲区取走一组类型 k 的材料;
        V(mutex);
        V(empty);              // 材料离开缓冲区，立即释放槽位

        卷烟并抽掉;
    }
```

`P(empty)` 和 `P(offer[k])` 必须放在 `P(mutex)` 之前。否则进程可能在持有互斥锁时等待空槽或等待某种组合，使能够改变条件的其他进程无法进入缓冲区，进而造成死锁。

## 容量 N 到底统计什么

上面的 `empty` 统计的是**物理缓冲区槽位**，所以吸烟者取走材料后就应执行 `V(empty)`，卷烟和吸烟可以在缓冲区外进行。

如果题目规定**最多允许有 $N$ 组尚未完成的吸烟任务**，那么 `empty` 统计的是从生产到吸烟结束的在途任务数。此时应把 `V(empty)` 移到“卷烟并抽掉”之后。
